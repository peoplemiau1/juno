# Концепция безопасности Juno (Safety Checker & Affine Types)

Текущий анализатор безопасности Juno (`safety_checker.rb`) отслеживает состояния ресурсов локально на уровне графа потока управления (CFG). Для обеспечения абсолютной безопасности управления памятью и ресурсами в Juno внедряется концепция аффинных типов (affine types) и сопутствующие механизмы.

## 1. Аффинные типы (Move Semantics & Borrowing)

По аналогии с Rust, управление ресурсами строится на следующих правилах:

### Единственное владение (Single Ownership)
* Каждый ресурс (выделенная куча, открытый файл) имеет ровно одного владельца (переменную) в любой момент времени.
* При присваивании или передаче ресурса в функцию владение перемещается (Move Semantics).
* Использование исходной переменной после перемещения владения запрещается на этапе компиляции (Compile-time check).

### Правило заимствования (Aliasing XOR Mutability)
* Допускается неограниченное число неизменяемых (совместных) ссылок на ресурс (`&T`).
* Допускается ровно одна изменяемая (эксклюзивная) ссылка на ресурс (`&mut T`).
* Одновременное существование изменяемой ссылки и других ссылок (изменяемых или неизменяемых) запрещено. Это исключает гонки данных и невалидный доступ при параллельном выполнении или повторном входе.

### Время жизни ссылок (Lifetimes)
* Компилятор статически гарантирует, что ссылка не может пережить ресурс, на который она указывает (предотвращение Dangling Pointers).

---

## 2. Интеграция SMT-солверов (Refinement Types)

Для перехода от простых типов данных к типам с уточнениями (Refinement Types), Juno использует математические спецификации прямо в сигнатурах:

```juno
fn divide(a: int, b: int {b != 0}): int
fn get_element(arr: Array, idx: int {0 <= idx && idx < len(arr)})
```

Интеграция SMT-солвера (например, Z3) на этапе компиляции позволяет математически доказать:
1. **Отсутствие деления на ноль**.
2. **Безопасность доступа к массивам** (отсутствие out-of-bounds) без рантайм-проверок.
3. **Отсутствие целочисленных переполнений**.

---

## 3. Потокобезопасность на уровне типов (Data Race Prevention)

Для предотвращения гонок данных вводятся маркерные интерфейсы (подобные Send и Sync):

* **Send**: Маркирует типы, которые можно безопасно передавать в другие потоки выполнения (отсутствие общих несинхронизированных ссылок).
* **Sync**: Маркирует типы, к которым можно осуществлять доступ из нескольких потоков параллельно (неизменяемые структуры или типы, защищенные примитивами синхронизации вроде мьютексов).

Передача объектов, не реализующих Send, в новые потоки (через `thread_spawn`) блокируется компилятором.

---

## 4. Отказ от Undefined Behavior (UB) в пользу Panic

Все ситуации, безопасность которых не может быть доказана статически, переводятся в контролируемые паники:

* **Проверка границ (Bounds Checking)**: Если компилятор/SMT-солвер не может статически гарантировать корректность индекса массива, генерируется неявная рантайм-проверка. Выход за границы приводит к завершению работы программы (panic) вместо чтения/записи произвольной памяти.
* **Инициализация памяти**: Вся выделяемая память стека и кучи автоматически заполняется значениями по умолчанию (например, нулями) для предотвращения чтения остаточного мусора.
